## Final Android Manifest Permissions

Declared in `android/app/src/main/AndroidManifest.xml`:

1. `android.permission.INTERNET`
Reason: Supabase APIs, WhatsApp OTP edge function traffic, image/PDF uploads, Firebase Crashlytics transport.

2. `android.permission.ACCESS_NETWORK_STATE`
Reason: Required by the Wi-Fi auto-connect path and general network state checks.

3. `android.permission.CHANGE_NETWORK_STATE`
Reason: Required by the Android Wi-Fi auto-connect flow based on `WifiNetworkSpecifier` / `NetworkRequest`.

4. `android.permission.ACCESS_FINE_LOCATION`
Reason: Required by the current Wi-Fi plugin implementation for Android SSID access and app-scoped Wi-Fi connection. The app does not use location for venue discovery, ads, or analytics.

5. `android.permission.NEARBY_WIFI_DEVICES`
Reason: Required on Android 13+ for managing nearby Wi-Fi connections. Declared with `neverForLocation`.

6. `android.permission.ACCESS_WIFI_STATE`
Reason: Required to read the current Wi-Fi connection state/SSID during the guest venue Wi-Fi flow.

7. `android.permission.POST_NOTIFICATIONS`
Reason: Required on Android 13+ so venue staff can receive operational push notifications for new orders and bell requests.

### Generated Internal Permission In The Merged Manifest

The merged Android manifest also contains `com.dineinmalta.app.DYNAMIC_RECEIVER_NOT_EXPORTED_PERMISSION`, which is generated by AndroidX Core for internal receiver protection. It is not a runtime user permission and does not create an additional Play Console sensitive-permission declaration.

## Permissions Intentionally Not Declared

These were removed to reduce Play review surface and avoid unnecessary declaration burden:

1. `android.permission.CAMERA`
Not needed on Android because venue photo capture uses the system camera intent through `image_picker`, not direct camera APIs.

2. `android.permission.READ_MEDIA_IMAGES`
Not needed because venue uploads use system pickers instead of broad media library access.

3. `android.permission.READ_EXTERNAL_STORAGE`
Not needed because venue uploads use system pickers instead of broad filesystem/media access.

4. `android.permission.CHANGE_WIFI_STATE`
Not needed because the app does not directly toggle device Wi-Fi state.

## Runtime Permission Behavior

1. Guest startup:
The app shows a custom `LOCATION SHARING` disclosure popup on guest routes. Its CTA deep-links to app settings and the copy now matches the real feature scope: venue Wi-Fi connection only.

2. Guest Wi-Fi auto-connect:
When a guest explicitly taps the venue Wi-Fi action, the app requests the Android runtime permissions actually needed by the feature:
- `ACCESS_FINE_LOCATION`
- `NEARBY_WIFI_DEVICES` on Android 13+

3. Venue camera capture:
Android uses the system camera intent and does not request the Android `CAMERA` permission.

4. Venue photo/PDF upload:
Android uses system pickers and does not request broad media/file permissions.

5. Venue operational alerts:
When a venue staff session is active, Android 13+ may request the notification runtime permission so the app can deliver new-order and bell-request push alerts.

## iOS Permission Posture

The iOS app is aligned to the same minimum scope:

1. `NSLocationWhenInUseUsageDescription`
Used only for guest venue Wi-Fi connection support.

2. `NSLocalNetworkUsageDescription`
Used by the iOS Wi-Fi join flow.

3. `NSCameraUsageDescription`
Used only when venue users capture menu photos during onboarding.

The CocoaPods permission macros are configured for `locationWhenInUse` only and camera only. No photo library permission is enabled because the app uses system pickers instead of broad library access.

## Play Console: Expected Declaration Scope

### Permissions Declaration Form

With the current Android manifest, the app should not trigger a special broad photo/video access declaration, because the app no longer declares `READ_MEDIA_IMAGES` or `READ_EXTERNAL_STORAGE`.

If Google Play surfaces any additional declaration questionnaire, review it carefully in the current console flow before submitting. Policy screens can change independently of the app code.

### Data Safety Section

The following data flows should be reviewed and declared in Play Console:

1. `Phone number`
Collected:
- Venue/admin WhatsApp OTP login and verification
- Venue claim / contact workflows

2. `Photos`
Collected:
- Venue-captured menu photos uploaded for OCR/menu setup

3. `Files and docs`
Collected:
- Venue-uploaded PDFs and menu image files uploaded for OCR/menu setup

4. `Crash logs`
Collected:
- Firebase Crashlytics in release builds

5. `Diagnostics`
Collected:
- Firebase Crashlytics runtime diagnostics and stability metadata

6. `Device or other IDs`
Collected:
- Firebase Crashlytics identifiers as documented by Firebase's Play disclosure guide
- Firebase Cloud Messaging registration tokens used for venue operational push alerts

### Data Not Collected Off Device

1. `Exact location`
The app requests location for guest Wi-Fi auto-connect, but the current app flow does not transmit device location off the device. Based on the Play Data safety guidance, on-device-only access should not be disclosed as collected user data. The privacy policy should still explain why the permission is requested.

## Privacy Policy Copy To Include

At minimum, the public privacy policy used in Play Console should include:

1. DineIn uses device location only to help guests connect to venue Wi-Fi and does not use that location for advertising or venue discovery.

2. Venue users can upload menu photos and PDF documents. Those files are sent to DineIn infrastructure for storage and menu extraction/OCR.

3. Venue and admin phone numbers are used for WhatsApp OTP authentication and account verification.

4. Release builds use Firebase Crashlytics for crash reporting and stability diagnostics.

5. Venue staff devices may store Firebase Cloud Messaging registration tokens so DINEIN can send new-order and table-alert push notifications.
